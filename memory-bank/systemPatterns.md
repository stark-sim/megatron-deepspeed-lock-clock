# System Patterns

## Network-Aware Prediction Pattern
- [2026-04-09] **Continuous Benchmark-Scaled Penalty Coefficients**: `torch_nccl_comm_bench.py` should no longer drive a hard `fast-vs-slow` branch. Instead, preserve the learned TP/PP/DP exposed-time structure, then continuously rescale `alpha_*` / `beta_pp_*` using benchmark-derived communication speed and jitter so the penalty parameters move smoothly across Ethernet / RoCE / IB.
- [2026-04-09] **Benchmark-As-Parameter-Translator**: The all-reduce benchmark is not itself the final communication penalty. Its role is to estimate the current topology's transport speed, then translate that speed into adjusted cross-node penalty coefficients that are consumed by the existing additive penalty model.
- [2026-04-09] **Size-Aware Network Translation**: When benchmark JSON includes multiple message sizes, use the measured size/bandwidth curve to estimate representative transport speed for the current penalty family instead of collapsing the whole network into one fixed scalar. This keeps the predictor topology-aware while avoiding a direct "benchmark time = penalty time" shortcut.
- [2026-04-09] **Cluster-Capacity Transfer Scaling**: When source and target topologies use different total GPU counts, the base throughput anchors must not stay frozen at the source cluster size. Scale base compute/memory capacity by `(target_total_gpus / reference_total_gpus) * (target_pipeline_parallel_efficiency / reference_pipeline_parallel_efficiency)` so transfer captures the larger cluster's aggregate capacity while still discounting lower pipeline utilization.
- [2026-04-09] **Throughput vs Power Cluster Scope**: The new cluster-capacity scale belongs in global throughput prediction, but not in the power utilization denominator. In this project the compared `avg_power_w` behaves like per-node power, so power should keep local-node compute normalization even when throughput is scaled to a larger cluster.
- [2026-04-11] **Per-Node GPU-Count Power Scaling**: When `avg_power_w` reflects per-node aggregate power rather than whole-cluster power, transfer must explicitly scale power by `target.gpus_per_node / reference_gpus_per_node`. This scaling is separate from cluster-capacity throughput scaling: total-GPU growth explains throughput, while local-GPU growth explains the per-node power magnitude. Keep the existing topology/cross-node power modifiers for shape, but anchor the main magnitude with local GPU count first.
- [2026-04-03] **Cross-Network Transfer**: A model calibrated on one network type (e.g., IB) can predict on another (e.g., RoCE) by supplying the target network's benchmark results during prediction time.

## Cross-Hardware Unified Prediction Pattern
- [2026-04-28] **Single Physics Engine, Hardware-Specific Calibration**: The same `analysis.freq_model` API (`predict_point()`, `predict_throughput_tokens_per_s()`, `predict_power_w()`) should drive predictions for all hardware platforms. Differences between RTX 4080S (Ethernet, comm-bottleneck) and V100 (NVLink, compute-bottleneck) are captured entirely by `CalibrationParams`, not by branching to different predictor implementations. The "analytical fit" shortcut (`T(f)=a·(f_max/f)^b+c`) should be treated as a post-hoc verification tool, not as a production prediction path.
- [2026-04-28] **Physics-Driven Calibration Parameter Derivation**: `CalibrationParams` must be DERIVED from hardware specs + workload features + network config, not hard-coded per scenario. The derivation layer (`derive_calibration_params()`) computes:
  - `compute_limit` from Roofline model: `peak_tflops * gpu_count * 1e12 / flops_per_token * compute_efficiency * min(1, AI/hardware_balance)`
  - `memory_limit` from memory bandwidth: `mem_bw_gbps * gpu_count * 1e9 / bytes_per_token * memory_efficiency`
  - `communication_limit` from network bandwidth: `tokens_per_step / (exposed_comm_bytes / net_bw / network_efficiency)`
  - `communication_penalty` from transport type (NVLink/IB/Ethernet/PCIe mapping)
  - `static_power_w` and `dynamic_power_w` from `HardwareFingerprint` (calibrated, NOT derived from TDP, because training power is far below thermal design power)
  This makes the predictor generalizable: changing the model (Qwen2-0.5B) or topology (dual-node 2×2) automatically updates the limits without new code.
- [2026-04-28] **HardwareFingerprint: One Platform, One Calibration**: Introduce `HardwareFingerprint` as the sole per-hardware-platform calibration artifact (~8 parameters). It is calibrated once from observed sweeps on that platform and reused for any model or topology. The fingerprint contains:
  - Efficiency factors: `compute_efficiency`, `memory_efficiency`, `network_efficiency`
  - Power-model params: `static_power_w`, `dynamic_power_w`, `dynamic_power_exponent`, `power_utilization_exponent`
  - Thermal params: `thermal_throttle_threshold`, `thermal_throttle_coefficient`
  Calibrate via `calibrate_hardware_fingerprint()` (grid search over fingerprint space + correction-layer refinement), then apply to new workloads via `derive_calibration_params()`.
- [2026-05-06] **HardwareFingerprint Utilization-Regime Dependency** (empirically discovered via Qwen2.5-1.5B @ 2×2 validation): The `static_power_w` and `dynamic_power_w` parameters in `HardwareFingerprint` are NOT pure hardware constants. They are **regime-dependent** on GPU utilization:
  - High-utilization regime (Qwen7B @ 2×4, 8 GPUs): `static=212.7W`, `dynamic=161.3W` fits well. Actual power range: ~214W→309W per node.
  - Low-utilization regime (Qwen1.5B @ 2×2, 4 GPUs): Same fingerprint predicts baseline 286W vs actual 321W (+12%) and static2505 308W vs actual 258W (-16%). Actual power range: ~110W→129W per node — much flatter than predicted.
  - Root cause: `power = static + dynamic * (utilization ** pue) * (f/f_max)**exp` assumes `static`/`dynamic` are fixed hardware properties, but actual idle and loaded power both decrease when GPU utilization drops because fewer SMs are active, less memory traffic, and less interconnect activity.
  - **Implications**:
    1. "One fingerprint per hardware platform" is an approximation, not a rigorous invariant. The approximation works well when the calibration and prediction workloads have similar utilization (~same model size relative to GPU capacity).
    2. For large utilization gaps (e.g., 7B→1.5B on same hardware), either accept ~15-20% power MAPE as the generalization cost, or introduce a **utilization-scaling correction**.
    3. A possible correction: `power_actual = power_fingerprint * utilization_correction(utilization)`, where `utilization_correction` is a learned or physically-motivated function (e.g., `α + (1-α) * utilization ** β`) that scales both static and dynamic components.
    4. `power_utilization_exponent` alone is insufficient because it only scales the dynamic component by utilization; it does not capture that the static floor itself drops at low utilization.
  - **Decision**: For now, document this as a known limitation. The ~10% target MAPE was designed for "same utilization regime" transfers. Small-model predictions (where utilization is low) will have higher power error. This is acceptable for the "generalization-first" design philosophy.
- [2026-04-28] **Power-Utilization Decoupling Parameter**: When communication dominates (4080S Ethernet), GPU power consumption is frequency-dominated and largely independent of actual compute throughput. Add `power_utilization_exponent` to `CalibrationParams` to control the coupling strength between `utilization = throughput / compute_limit` and dynamic power. `exponent=0.0` makes power purely frequency-driven; `exponent=1.0` restores the original `dynamic_power_w * utilization * freq_ratio^exp` behavior. This single parameter unifies both regimes under one `predict_power_w()` formula.
- [2026-04-28] **Max-Frequency Semantics Separation**: `HardwareFeatures.max_frequency_mhz` has different semantics for static vs baseline prediction. Static locking uses the experimental locking ceiling (e.g., 2505 MHz for 4080S), while baseline dynamic boost uses the theoretical GPU boost ceiling (e.g., 3105 MHz). The thermal throttle factor `_thermal_throttle_factor()` computes `frequency_ratio = nominal_freq / max_frequency`, so using the wrong ceiling produces wildly incorrect θ values (0.500 vs 0.918). Maintain two `HardwareFeatures` instances or override `max_frequency_mhz` at prediction time depending on `mode`.
- [2026-04-28] **Cluster-Capacity Scale Consistency Rule**: When calibrating on one workload and predicting on another, the `reference_pipeline_parallel_efficiency` field in `CalibrationParams` must be set consistently. If the calibration script omits it (default 1.0), the prediction script must also omit it; otherwise `cluster_capacity_scale` changes and predictions diverge by the pipeline-efficiency ratio (~0.667 in this project).

## Hardware-Workload Separation Pattern
- [2026-04-28] **Workload-Agnostic Calibration, Workload-Aware Prediction**: The predictor must not gate calibration or prediction on model identity ("Qwen" vs "LLaMA"). Instead, all experiment samples sharing the same hardware and topology should be pooled into a single calibration run regardless of model name. The resulting calibrated hardware parameters (`b`, `exp`, `P_static`, `P_dynamic`) are then applied to any workload by using that workload's specific `DerivedModelFeatures` (param count, FLOPs/token, attention shape, pipeline/TP/DP exposure, etc.) at prediction time. This makes the predictor architecture-agnostic: Qwen and LLaMA on the same hardware get the same hardware parameters but different curves because their workload-derived features differ.
- [2026-04-28] **GQA-Aware Attention Parameter Accounting**: `derive_model_features()` must account for Grouped Query Attention when estimating attention parameter count. The correct formula is `approx_attention_params = hidden_size² × (2 + 2 × kv_ratio)` where `kv_ratio = num_key_value_heads / num_attention_heads`. Using the naive `4 × hidden_size²` overestimates GQA models (e.g., Qwen 28/4 by ~75%) and causes incorrect `approx_model_params` / `approx_flops_per_token` downstream. This distortion propagates into `arithmetic_intensity`, `compute_weight`, and ultimately throughput predictions, so fixing it is prerequisite for fair cross-model prediction.
- [2026-04-28] ~~**Hardware-First Parameter Invariance**~~ → **DEPRECATED** (superseded by Physics-Driven Derivation). The old pattern stated that `b`/`exp`/`P_static`/`P_dynamic` remain invariant across models on the same hardware. This was correct in spirit (hardware characteristics dominate) but wrong in implementation (it encouraged hard-coding these values). The new approach: `HardwareFingerprint` is the only per-hardware artifact; `derive_calibration_params()` automatically computes model-specific limits from `DerivedModelFeatures`. No manual "borrowing" or "re-fitting" is needed.
- [2026-04-28] ~~**Compute-Communication Workload Decomposition**~~ → **DEPRECATED** (superseded by Physics-Driven Derivation). The old pattern stated that `a`/`c` are the primary workload-differentiating knobs and should be re-fitted for new models. This is no longer necessary: `derive_model_features()` already computes FLOPs/token, memory bytes, and cross-node communication bytes from model structure. The "workload differentiation" happens automatically in the derivation layer.
- [2026-04-28] ~~**Two-Point Hardware-Prior Calibration**~~ → **DEPRECATED** (superseded by HardwareFingerprint). The old pattern stated that two observed points are sufficient for a new model by borrowing `b`/`exp` from prior calibrations. The new approach: zero points are needed for a new model on an already-characterized hardware platform. Simply change the `WorkloadFeatures` (num_layers, hidden_size, etc.) and `derive_calibration_params()` produces the full prediction curve. The `HardwareFingerprint` has already absorbed all hardware-specific calibration from prior sweeps.
- [2026-04-28] **Hardware-Parameter Non-Identifiability On Limited Band**: When the observed frequency band is narrow (e.g., 1200–2505 MHz, ratio <2.1×), the time-model exponent `b` and power-model exponent `exp` are not independently identifiable from the data alone. Different `(b, exp)` pairs can achieve identical prediction accuracy (same MAPE) by compensating through the additive parameters (`a`, `c`, `P_static`, `P_dynamic`). On LLaMA-7B 32L (4 points), independent fit gives `b=0.754`/`exp=5.731` while Qwen hardware prior gives `b=1.046`/`exp=8.0`, yet both models yield the same 2.16% time MAPE. Therefore: (1) for prediction accuracy, either model works; (2) for claiming "hardware invariance" of `b`/`exp`, one needs either a much wider frequency band (≥3× ratio) or a dedicated micro-benchmark that isolates compute scaling from communication overhead. **Note**: This identifiability problem is now moot for new-model prediction because `derive_calibration_params()` does not require fitting `b`/`exp` from new data.

## Predictor Generalization-First Pattern
- [2026-04-28] **Generalization Over Per-Model Accuracy**: The predictor's primary goal is **generalization** (predict correctly for any new model/topology on characterized hardware), not **per-model MAPE minimization**. When these two goals conflict, generalization wins. Concretely:
  - `calibrate_hardware_fingerprint()` performs **joint calibration** across all models on the same hardware (e.g., pool Qwen + LLaMA samples for 4080S). It does NOT optimize a separate fingerprint per model, even though a per-model fit would yield lower MAPE.
  - `derive_calibration_params()` uses physics-derived formulas (Roofline, bandwidth models) for compute/memory/communication limits. It does NOT distort these limits to fit a single model's observed data better.
  - A joint-calibration MAPE of ~10% is **acceptable** because it buys the ability to predict new models (Qwen2-0.5B, DeepSeek-7B, etc.) without any recalibration.
  - The old v2 approach achieved 0.83% MAPE on 4080S Qwen by hand-tuning `compute_limit=7000`, `memory_limit=600`, `comm_limit=300` to exactly match that model's data. That approach is **intentionally abandoned** in v3: those hand-tuned values are not physically derivable and do not generalize.
  - **Decision rule**: If someone proposes "let's add a per-model correction factor to squeeze MAPE from 10% to 3%", the answer is NO — unless that factor can itself be derived from model structure (e.g., from `num_layers`, `hidden_size`, `attention_type`) without observed data. The predictor must remain zero-shot for new models.
- [2026-04-28] **Physics-Derived Limits Are Truth, Not Fitting Targets**: The three bottleneck limits (`compute_limit`, `memory_limit`, `communication_limit`) in `CalibrationParams` are derived from first principles and should NOT be hand-tuned to improve MAPE on any specific model. If the predicted curve deviates from observed data, the correct response is to check:
  1. Is the `HardwareFingerprint` calibrated on enough diverse models?
  2. Are there missing features in `derive_model_features()` (e.g., embedding layer FLOPs, MoE routing overhead)?
  3. Is the correction layer (`_fit_curve_corrections`) underfitting due to insufficient search granularity?
  **Wrong response**: "Let's tweak `compute_limit` from 2883 to 7000 because that makes Qwen fit better." That destroys physical interpretability and generalization.

## Communication-Model Validation Pattern
- [2026-04-21] **Real-Qwen V100 Topology-Specific Launcher Pattern**: 对 `DGX2-1 + DGX2-2` 的真实 `Qwen2.5-7B-Instruct` 双机训练，不再把 driver 藏在 `.context/`。当前规范入口已按拓扑拆分到 `scripts/`：`scripts/run_real_qwen25_7b_tp2pp2dp2_v100.sh` 对应 `TP=2 / PP=2 / DP=2`，`scripts/run_real_qwen25_7b_tp4pp2dp1_v100.sh` 对应 `TP=4 / PP=2 / DP=1`。此外，`scripts/run_real_qwen25_7b_tp4pp2dp1_v100_compare.sh` 会串行执行 `baseline / static1395 / static1252 / static1155`。这些 launcher 只负责固定 hostfile、`DS_INCLUDE`、`LOCAL_GPU_INDICES=8,9,10,11`、IB 网络环境、真实 checkpoint `--load` 路径与 `DISABLE_SAVE_CHECKPOINT=1`，实际工件仍统一落在 `experiments/<run_id>/`。
- [2026-04-11] **Primary-Node Artifact Authority Rule For DGX2 Formal Runs**: For the dual-node IB jobs in this project, treat `DGX2-1` (the launcher / node0 side) as the source of truth for formal artifact verification. `DGX2-2` may only retain local fragments such as `ds_config.json`. To confirm that a result is not a local parsing hallucination, verify on `DGX2-1` that: (1) `hostfile_snapshot.json` includes both `v100x16-1` and `v100x16-2`, (2) `run.json/topology/preflight` match the intended GPU slice and world size, and (3) the primary training log contains `v100x16-2:`-prefixed iteration lines plus Zeus final summaries.
- [2026-04-11] **Topology-Fixed Replay Rule For World-Size Transfers**: Do not use `scripts/evaluate_transfer_prediction.py` as-is for canonical world-size transfer checks such as `2x4 -> 2x8` when `data_parallel_size` and `global_batch_size` intentionally change with topology. The stock CLI currently treats that as a workload mismatch. Use a topology-fixed replay instead: calibrate on the source samples, evaluate against target workload features, and record explicit `source_override` / `target_override` in the output artifact so the transfer remains auditable.
- [2026-04-10] **Transport-Provenance Pairing Rule**: Do not combine a benchmark JSON and a source/target training-artifact pair into one "formal IB/Ethernet result" unless both were collected under the same verified transport configuration. A separate single-node benchmark can still be useful for sensitivity analysis, but if the paired training artifacts came from a different launcher path or different `NCCL_*` env, the result must be labeled mixed-provenance / provisional rather than formal.
- [2026-04-09] **Reconstructed-Artifact Sanity Check Rule**: If the exact historical transfer pair is cited in notes or the paper draft but the current workspace snapshot no longer contains the raw target artifacts, do not silently treat derived numbers as a formal replay. It is acceptable to do a reconstructed sanity check by combining documented observed points with still-available topology definitions, but the result must be labeled reconstructed/provisional and cannot replace the paper's main accuracy claim until the raw target artifacts are recovered or rerun.
- [2026-03-27] Communication-model validation should prioritize `2x4 -> 2x8` transfer before larger world-size jumps; avoid using `2x8 -> 2x16` as the first regression check when iterating on communication accuracy.
- [2026-03-26] Pooled multi-topology calibration should not rely only on fitted residual sensitivities. If explicit `base_topology_*_sensitivity` terms converge to zero, keep a dispersion-scaled topology prior active in the base model itself so held-out topologies still inherit a topology-conditioned throughput/communication/power/shape bias. In this project that shift materially improves held-out `tp4pp1dp4`, even though the residual search still chooses zero additional coefficients.
- [2026-03-26] If pooled multi-topology calibration still picks `base_topology_*_sensitivity = 0` and held-out error stays unchanged, treat that as evidence that the topology multipliers are not merely underfit but structurally too weak. In that case, move the next iteration earlier into the base model: topology-condition the anchor / compute-frequency elasticity itself, or learn pooled anchors keyed by topology pressure, instead of stacking more correction-layer damping or small post-hoc scales.

## Experiment Execution Pattern
- Launcher entry point: `scripts/run_experiment.sh`.
- Shell helpers: `scripts/experiment_utils.sh`.
- Training/power integration: `megatron/training.py` and `megatron/power_monitor.py`.
- Experiment metadata is stored in `experiments/<run_id>/` as JSON, JSONL, notes, command snapshots, and logs.
- [2026-04-23] **Zeus Raw-Metrics Rule**: `megatron/power_monitor.py` 中的 Zeus interval summary 必须保留原始真实口径；不要再根据 `EXPERIMENT_MODE=static` 对 `time_s`、`avg_power_w`、`energy_j` 做二次缩放。若需要展示层修正或对比层归一化，应放在离线分析/汇报脚本中，而不是写回训练时落盘的正式工件。

- [2026-04-19] **Real-Qwen HF->Megatron Bring-Up Pattern On `sd-1/sd-2`**: 对 `Qwen2.5-7B-Instruct` 这类 HF 权重主线，先在每台 Ethernet 节点本地使用 `4 GPUs` 做一次独立 `hf2megads` 转换，再进入双节点训练；当前验证过的安全拓扑是 `TP=2 / PP=2 / DP=1`。转换器不能再把 `tokenizer.vocab_size` 当作唯一词表真值，而应优先信任 HF `model.embed_tokens.weight.shape[0]`，并在构建 Megatron/DeepSpeed 模型前按 HF `config.json.vocab_size` 上调 `padded_vocab_size`。产物验收信号是：日志出现 `save checkpoint completed`、目录包含 `latest=global_step0`、`global_step0` 下有完整 layer shards / `mp_rank_*` 文件。完成双机独立转换后，再把两端 checkpoint 规范到相同逻辑路径，进入真实 checkpoint `--load` 训练 smoke。

In this project, `TP=4` combined with `kv_heads=2` does not fail until `ParallelAttention` construction (`AssertionError: 2 is not divisible by 4`), which wastes a full distributed bring-up cycle. `scripts/run_experiment.sh` should fail fast on that incompatibility so topology sweeps only spend remote time on legal model shapes.

Because each host has its own `/dev/shm`, one node cannot rely on another node's freshly generated mmap index cache. For standard dual-node Megatron runs in this project, prefer per-node persistent `data/index-cache` and verify the same `<hash>_{doc,sample,shuffle}_idx.npy` files exist on all participating hosts before launch.

Setting `TEXMFVAR`, `TEXMFCONFIG`, and `VARTEXFONTS` under `.context/paper/.texlive-cache/` avoids sandbox/home-directory write issues and makes the build path more reproducible across shells.

Launch a remote driver script on `DGX2-1` via `nohup`/`screen`, let it own the sweep, and then monitor logs separately. If the local PTY dies mid-run, DeepSpeed/pdsh workers can remain stranded across both nodes and require manual cleanup before the next experiment.
- [2026-04-10] **Per-Node Mmap Index-Cache Consistency Rule**: before increasing node count or GPU count for a Megatron mmap workload, confirm every node has the exact `data/index-cache/<hash>_{doc,sample,shuffle}_idx.npy` files for that workload. ## Frequency-Lock Pattern
- `EXPERIMENT_MODE=baseline` keeps default GPU behavior.
- `EXPERIMENT_MODE=static` with `STATIC_CLOCK_MHZ=<mhz>` uses NVML clock locking.

## Energy-Only Sweep Pattern
- Short sweep runs use a fixed topology and training length to isolate clock effects.
- `DISABLE_CHECKPOINT=1` is preferred for sweep jobs because checkpoint artifacts are large and unnecessary for energy analysis.
- Zeus power monitoring is used to generate energy, average power, `samples/Wh`, and `tokens/J`.

## Comparison Pattern
- Keep model, tokenizer, data path, TP/PP/DP, ZeRO stage, and train steps constant across variants.
- V100 short-run sweep baseline uses `TP=4`, `PP=1`, effective `DP=4`, `ZeRO-1`, `TRAIN_STEPS=20`.
- Use log-derived steady-state iteration time plus Zeus metrics for direct comparison.
- [2026-03-15] Clock cleanup for experiment runs should be layered: keep the in-process NVML reset in `megatron/gpu_freq_manager.py`, but also add a launcher-level `EXIT` trap in `scripts/run_experiment.sh` for any mode that may touch clocks (`static`, `dynamic`, `dryrun`) so node-local GPUs are unlocked even if the training process exits without reaching the normal Python shutdown path. Keep this path quiet on success and warn only on reset failures unless actively debugging clock cleanup.

## Artifact Pattern
- `run.json` is the canonical run summary when finalize succeeds.
- `events.jsonl` is a reliable backup source for interval metrics, especially when final checkpointing or finalize logic is incomplete.
- The training log remains the fallback truth source for final `[Zeus] Steps 1-20 ...` summaries.

## Predictive Modeling Pattern
- [2026-04-28] **Baseline/Static Dual-Mode Prediction Pattern**: The predictor must expose two distinct prediction modes:
  - `mode='static'`: Fixed-frequency prediction **without** thermal throttling. Static locking stabilizes GPU temperature (65–70°C observed on 4080S), eliminating driver-level frequency throttling and multi-GPU desynchronization. The `_thermal_throttle_factor` is bypassed.
  - `mode='baseline'`: Dynamic-boost prediction **with** thermal throttling. Baseline nominal frequency (e.g., 2505 MHz on 4080S) triggers thermal throttling because actual GPU temperature exceeds the driver's throttle threshold. Additionally, each GPU throttles independently, creating multi-GPU desynchronization that adds NCCL wait overhead. Both effects are approximated by `_thermal_throttle_factor(frequency_ratio)`.
  - **API contract**: `predict_throughput_tokens_per_s`, `predict_power_w`, `predict_point`, and `sweep_prediction_points` all accept `mode='static'` (default) or `mode='baseline'`. Calibrate static curves against static sweep data; calibrate baseline thermal parameters against actual baseline runtime measurements.
  - **Physical justification**: The static time model `T_static(f) = a·(f_max/f)^b + c` does **not** contain θ(f). The baseline runtime is `T_baseline = T_static(f_nominal) + ΔT_thermal + ΔT_desync`, where ΔT_thermal is the slowdown from effective-frequency drop and ΔT_desync is the slowdown from waiting for the slowest throttled GPU. The predictor merges both into a single multiplicative `θ(frequency_ratio)` for simplicity, but the coefficient must be calibrated against baseline measurements, not static sweep data.
- [2026-04-28] **Thermal Throttle Coefficient Calibration Rule**: Do **not** fit the thermal throttle coefficient from static sweep data. Static sweep data by design has minimal thermal throttling because temperature stabilizes after the first few steps. Instead:
  1. Collect a true baseline measurement (dynamic boost, no clock lock).
  2. Collect a static measurement at the same nominal frequency (if the hardware supports locking at that frequency) or at the nearest supported point.
  3. The thermal throttle factor is inferred from the ratio `throughput_static / throughput_baseline` at comparable frequencies.
  4. For 4080S LLaMA dual-node, the inferred combined penalty (thermal + desync) is ~8% (θ≈0.92), yielding coefficient≈0.65 with threshold=0.7. This is much larger than the ~2.5% penalty suggested by static-only fitting because static data cannot observe desynchronization overhead.
