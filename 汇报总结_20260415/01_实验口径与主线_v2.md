# 实验口径与主线（v2 修订版）

## 一、我们到底在比较什么

这里的核心比较不是"降频"和"默认频率"的泛泛对比，而是：

- `baseline`：Megatron-DeepSpeed 的原始默认运行方式，不额外锁频。GPU 在训练过程中根据温度动态调整频率，可能出现热节流（thermal throttling）和多卡不同步。
- `static`：在相同 workload、相同拓扑、相同训练步数下，把 GPU 固定到某个目标频点。消除动态频率的不确定性，温度稳定，多卡完全同步。

因此，这是一组受控对照实验，核心发现是：**baseline 动态频率的不稳定性导致的隐性性能损失**。

## 二、为什么 Static 中频能战胜 Baseline 高频

### 物理机制

```
Baseline（名义 2505 MHz）：
  高功耗 → 温度飙升 → 驱动自动降频至 2100-2300 MHz
       → 8 卡降频节奏不同步 → NCCL 等待最慢卡
       → 有效 step time = 计算 + 通信 + 同步等待 + 降频恢复延迟

Static 1800 MHz：
  中等功耗 → 温度稳定在 65-70°C → 从不降频
       → 8 卡频率完全一致 → NCCL 无额外等待
       → 有效 step time = 纯计算 + 通信（可预测）
```

### 实验证据

| 平台 | Baseline | Static 中频 | 结果 |
|------|----------|-------------|------|
| 4080S LLaMA | 2505 MHz, 281.0s | 1800 MHz, **276.1s** | **中频反而快 1.7%** |
| V100 Qwen | 1380 MHz, 724.6s | 1260 MHz, **717.6s** | **中频反而快 1.0%** |
| V100 LLaMA | 1380 MHz, 659.0s | 1260 MHz, **620.5s** | 中频快 5.8% |

**结论**："中频比高频快"不是计算奇迹，而是 baseline 动态频率不稳定性的直接暴露。

## 三、公平对比的最低要求

下面这些条件必须保持一致：

- 模型结构
- tokenizer
- 数据集
- `TP / PP / DP`
- 节点数与 GPU 选择
- micro batch / global batch
- 训练步数
- ZeRO 配置
- precision 模式

真正允许变化的只有：

- `EXPERIMENT_MODE`
- `STATIC_CLOCK_MHZ`

## 四、Zeus 在这里扮演什么角色

Zeus 是统一的观测口径，用来采集：

- 总训练时间
- 平均功率
- 总能耗

这样 baseline 和 static 的对比就不再依赖零散日志，而是落在统一指标上。

## 五、当前汇报的主线（v2）

最推荐的叙事顺序是：

1. baseline 是默认起点，但存在热节流导致的隐性性能损失
2. static 锁频消除不确定性，不仅省电，有时还能更快
3. 预测模型帮助用户快速定位最优频点，避免全频段盲扫
4. Zeus 提供统一证据链
5. 多平台验证：20%+ 能耗降低，时间损失 <3%

## 六、当前最稳妥的结论写法

- `在相同 workload 与拓扑下，将 GPU 固定在合适频点，可以在不显著增加训练时间的前提下，降低约 20% 量级的平均功率，部分拓扑更高。`
- `静态锁频的核心价值不仅是省电，更是消除了动态频率导致的温度抖动和多卡不同步，使训练行为变得可预测。`
- `预测层的作用是帮助用户缩小值得验证的频点范围，而不是替代真实实验。`

## 七、当前不建议的写法

- `预测层直接带来了 20% 节能`
- `所有拓扑都会落在同一个最佳频点`
- `所有历史 baseline 原始工件都已经完整保存在本地`
- `热节流是一个可以用数学公式完全拟合的现象`（它本质是动态运行时行为）
