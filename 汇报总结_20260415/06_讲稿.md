# 讲稿

## 30 秒版本

我们的 baseline 就是 Megatron-DeepSpeed 的原始启动配置，不做人为锁频；然后在完全相同的 workload 和拓扑下，把 GPU 固定到不同频点，用 Zeus 统一采集训练时间、平均功率和总能耗。结果表明，只要频点定得合适，训练总时间基本不需要付出明显代价，但平均功率可以下降约 20% 量级，部分拓扑更高。我们的预测层不是替代这个结论，而是帮助用户更快找到值得验证的频点。

## 90 秒版本

这项工作的出发点不是单纯证明“降频能省电”，因为那件事本身并不难。真正重要的是，在用户实际使用的 baseline 配置下，我们能不能找到一个固定频点，让训练总时间基本不变，同时把平均功率和总能耗明显压下来。为此，我们把 baseline 和 static 实验放在同一个 Megatron-DeepSpeed 启动框架里，只允许时钟策略不同，其余拓扑、batch、训练步数和数据路径全部保持一致，并使用 Zeus 做统一采集。已有代表性结果显示，在 V100 双机实验里，合适定频可以带来约 25% 到 35% 以上的平均功率下降，而 runtime 仍维持在 baseline 附近。这说明系统里确实存在一个值得利用的“合适频点区域”。我们的预测层，就是为了帮助用户更快找到这个区域，避免每次都从头人工扫频。

## 适合写进摘要的表述

Baseline 指的是 Megatron-DeepSpeed 的原始默认运行方式。在相同 workload 与相同拓扑下，将 GPU 固定在合适频点，可以在不显著增加训练时间的前提下，降低约 20% 量级甚至更高的平均功率。预测层的作用，是帮助用户高效定位这些候选频点，而不是替代真实实验。

## 答辩时适合强调的边界

- 我们证明的是 `存在有效定频区间`，不是 `任何低频都更优`
- 我们强调的是 `runtime 基本不变`，不是 `锁频一定更快`
- 预测层的价值是减少 sweep 成本，而不是绕过真实实验
