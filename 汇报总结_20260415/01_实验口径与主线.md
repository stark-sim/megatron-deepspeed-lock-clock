# 实验口径与主线

## 一、我们到底在比较什么

这里的核心比较不是“降频”和“默认频率”的泛泛对比，而是：

- `baseline`：Megatron-DeepSpeed 的原始默认运行方式，不额外锁频
- `static`：在相同 workload、相同拓扑、相同训练步数下，只把 GPU 固定到某个目标频点

因此，这是一组受控对照实验，而不是随意改动多项参数后的横向比较。

## 二、公平对比的最低要求

下面这些条件必须保持一致：

- 模型结构
- tokenizer
- 数据集
- `TP / PP / DP`
- 节点数与 GPU 选择
- micro batch / global batch
- 训练步数
- ZeRO 配置
- precision 模式

真正允许变化的只有：

- `EXPERIMENT_MODE`
- `STATIC_CLOCK_MHZ`

## 三、为什么这组比较有意义

用户实际使用时，最先接触到的一定是 baseline。它最容易启动，也最符合默认认知。但 baseline 不一定是能效最优点。

所以我们真正关心的问题不是：

- `能不能把频率调低`

而是：

- `能不能找到一个合适的固定频点，在不明显拖慢训练的情况下，降低功率和能耗`

这就是 baseline vs static 的价值。

## 四、Zeus 在这里扮演什么角色

Zeus 是统一的观测口径，用来采集：

- 总训练时间
- 平均功率
- 总能耗

这样 baseline 和 static 的对比就不再依赖零散日志，而是落在统一指标上。

## 五、当前汇报的主线

最推荐的叙事顺序是：

1. baseline 是默认起点
2. static 是公平对照
3. Zeus 提供统一证据
4. 合适频点确实存在
5. 预测层的价值是更快找到这个频点

## 六、当前最稳妥的结论写法

- `在相同 workload 与拓扑下，将 GPU 固定在合适频点，可以在不显著增加训练时间的前提下，降低约 20% 量级的平均功率，部分拓扑更高。`
- `预测层的作用是帮助用户缩小值得验证的频点范围，而不是替代真实实验。`

## 七、当前不建议的写法

- `预测层直接带来了 20% 节能`
- `所有拓扑都会落在同一个最佳频点`
- `所有历史 baseline 原始工件都已经完整保存在本地`
