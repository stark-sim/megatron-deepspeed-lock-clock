# 演示流程

## 0. 约定

- 默认从仓库根目录执行命令
- baseline 和 static 都复用同一条 `run_experiment.sh` 路径
- static 模式需要具备 NVML 锁频权限

## 1. 先做预检

```bash
CUDA_VISIBLE_DEVICES=0,1 \
VALIDATE_ONLY=1 \
bash 汇报总结_20260415/脚本/运行1p5b演示.sh
```

这一步会检查：

- 可见 GPU
- 数据集与 tokenizer
- launcher 是否可用
- static 模式下的锁频支持情况

## 2. 跑 baseline

```bash
CUDA_VISIBLE_DEVICES=0,1 \
EXPERIMENT_MODE=baseline \
TRAIN_STEPS=20 \
bash 汇报总结_20260415/脚本/运行1p5b演示.sh
```

## 3. 跑一个 static 点

```bash
CUDA_VISIBLE_DEVICES=0,1 \
EXPERIMENT_MODE=static \
STATIC_CLOCK_MHZ=1080 \
TRAIN_STEPS=20 \
bash 汇报总结_20260415/脚本/运行1p5b演示.sh
```

## 4. 跑一个小 sweep

```bash
CUDA_VISIBLE_DEVICES=0,1 \
STATIC_FREQS="1080 1200 1320" \
bash 汇报总结_20260415/脚本/批量运行1p5b对比.sh
```

这个脚本会：

1. 先跑一个 baseline
2. 再跑多个 static 频点
3. 在本目录下生成结果清单
4. 自动产出一张 Markdown 对比表

## 5. 单独生成对比表

```bash
python3 汇报总结_20260415/脚本/对比Zeus结果.py \
  experiments/<baseline_run_id> \
  experiments/<static_run_id_1> \
  experiments/<static_run_id_2>
```

## 6. 现场最适合展示什么

- `VALIDATE_ONLY=1` 通过
- baseline 启动命令
- static 启动命令
- 生成的 `run.json`
- 最后的 Zeus 对比表

## 7. 现场讲述建议

`这里我们没有改变模型、数据和拓扑，只改变 GPU 的时钟策略；因此最后 Zeus 给出的时间、平均功率和总能耗，可以作为 baseline 与 static 的直接公平对比。`
